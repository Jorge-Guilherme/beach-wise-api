### O que é nosso projeto?
Percebemos que muitas pessoas tinham uma visão um pouco distorcida do litoral pernambucano. Essas pessoas são turistas ou até mesmo moradores do próprio estado. Quando saímos atrás de informações para criarmos um app ou página web sobre o tema, observamos que esses dados estão dispersos. Resolvemos então criar uma API que reuna dados de blogs, foruns, portais de notícias confiáveis e até APIs públicas. Nosso objetivo é reuniar essas informações em uma só API, para que futuros desenvolvedores utilize-a em seus projetos.

### Quais funcionalidades disponibilizamos?
Visamos oferecer o máximo de informações possíveis. Através de nossos endpoints, iremos disponibilizar:
- Localização das principais praias para banhistas, prática de esportes e visitas norturnas;
- Informações sobre ataques de tubarão em tempo real;
- Principais pontos turísticos litorâneos;
- Hospedagens com integração do Airbnb e Booking;
- Eventos litorâneos atualizados;

### Arquitetura do SI
Aqui, entramos numa parte mais técnica. Explicaremos como funciona a arquitetura do nosso de sistema. Basicamente, estamos utilizando REST para nosso padrão de API. Escreveremos o código em Python com o auxílio do framework Flask. Além disso, usaremos a estrutura monolítica. Nossa API terá todos os endpoints documentados, explicando minuciosamente como utilizá-los.

### Gerenciamento do projeto
Iremos utilizar o SCRUM como métodologia de gerenciamento. As especificações do nosso SCRUM será:

- <b>Sprints de 1 semana -></b> A ideia é manter o grupo sempre entregando atividades no formato de micro-entregas.
- <b>Wekley Reports -></b> Faremos reuniões de alinhamento semanais.
- <b>Organização -></b> Para organizar nosso WorkFlow, estamos utilizando o Notion(Backlog do produto, Sprints Review, Sprints Retrospective, Backlog de Sprints, etc.).

### Envolvimento de usuárioss
Como estamos visando a entrega de uma API, nossos usuários serão os desenvolvedores. Inclusive, nós mesmo podemos ser futuros usuários, assim como qualquer pessoa que queira utilizar nossa API para criar uma aplicação. 

### Custos & Riscos:
Um dos principais riscos que observamos, é o tempo. Tendo em vista que temos um prazo apertado para a realização do projeto. Além disso, podemos considerar a utilização dos dados "públicos", considerando que algumas páginas nem sempre permitem a utilização 100% dos seus conteúdos. 

Em relação aos custos, como estamos utilizando o Docker, conseguimos hospedar gratuitamente(com algumas limitações que não atrapalham um MVP) no Fly.io. Nossos conteúdos que irão fornecer as informações, será 100% gratuito. Dessa forma, não teremos curstos com arquitetura, nem aquisição dos dados.
